## 1. Что включает базовая часть и какую проблему она решает

Под базовой частью TaskBridge я имею в виду уже работающий минимальный контур, который даёт пользователю практическую пользу без долгой настройки.

Сейчас в базовую часть входят:

- подключение системы и сохранение учётных данных
- добавление и хранение точек пользователя
- разовые free-text запросы по `stoplist` и `blanks`
- включение и выключение мониторингов из обычного текста без работы с ID
- доставка алертов и отчётов в Telegram
- упрощённый Telegram UX без обязательных slash-команд
- операторская наблюдаемость по мониторингам, логам и prod-состоянию
- регрессии на золотые сценарии приоритетного пользователя

Проблема, которую решает базовая часть:

- владельцу или управляющему неудобно постоянно вручную заходить в портал и проверять точки
- критичные сигналы легко пропустить среди рутины
- даже когда данные доступны, путь до ответа слишком длинный: зайти в систему, выбрать точку, выбрать отчёт, выбрать период, интерпретировать результат

Что TaskBridge закрывает уже сейчас:

- сокращает путь до ответа до одного короткого сообщения
- присылает критичное без ручной проверки портала
- держит под рукой состояние по точкам, а не только разовый отчёт

Для каких пользователей это особенно релевантно:

- владелец небольшой сети
- управляющий несколькими точками
- территориальный менеджер
- оператор, который следит за проблемами по точкам в течение дня

Когда проблема встречается часто:

- когда у пользователя несколько точек и проверка делается много раз за день
- когда есть регулярные stoplist или blanks-инциденты
- когда портал неудобен как оперативный интерфейс, а Telegram используется как рабочий канал

Для нашего priority user это не теоретическая проблема. У него уже есть регулярный operational use:

- повторяющийся stoplist по `Сухой Лог, Белинского 40`
- реальные critical blanks-инциденты, а не только тестовые проверки
- использование общего report-чата как рабочего контура

То есть базовая часть уже решает повторяющуюся ежедневную задачу, а не разовый demo-case.

## 2. Какие данные нужны для управления и как пользователь их использует

Ниже — минимальный перечень данных, без которых TaskBridge не сможет быть инструментом управления, а останется только пересылкой отчётов.

### 2.1 Данные по точкам

- список точек пользователя
- система, к которой относится точка
- чат доставки отчётов
- статус подключения

Зачем нужно:

- понимать, по каким точкам вообще можно запрашивать данные
- быстро строить запросы по одной точке и по всем точкам
- маршрутизировать отчёты в нужный чат

Типовые сценарии:

- `покажи стоп по всем точкам`
- `какие у меня подключены точки`
- `куда сейчас уходят отчёты`

### 2.2 Данные по stoplist

- позиции, которые сейчас в стопе
- новые позиции
- ушедшие позиции
- сколько дней позиция находится в стопе
- точка, категория и момент изменения

Зачем нужно:

- понимать не просто факт наличия стопа, а динамику
- быстро отличать новый риск от давно известной проблемы
- видеть, какие позиции уже вернулись в продажу

Типовые сценарии:

- утренняя проверка изменений
- разбор, что именно стало хуже с прошлого отчёта
- контроль хронических позиций в стопе

### 2.3 Данные по blanks

- точка
- период отчёта
- слот времени
- сервис
- категория
- severity: жёлтый или красный
- признаки новой/продолжающейся проблемы

Зачем нужно:

- отличать лимитное состояние от действительно критичного
- не слать красный alert на жёлтые зоны
- помогать пользователю понять срочность и масштаб проблемы

Типовые сценарии:

- `что сейчас красное`
- `покажи blanks за последние 3 часа`
- `есть ли по всем точкам красные зоны`

### 2.4 Данные по мониторингам

- какие мониторинги активны
- по какой точке и какому типу они включены
- интервал
- окно работы
- когда последний раз проверялось
- какой был последний статус
- был ли недавний alert

Зачем нужно:

- чтобы пользователь понимал, что у него уже автоматизировано
- чтобы можно было быстро выключить шумящий мониторинг или усилить полезный
- чтобы оператор мог разобрать продовое состояние без гадания

Типовые сценарии:

- `какие у меня мониторинги`
- `выключи blanks по этой точке`
- `оставь только критичное`

### 2.5 Данные о реакции пользователя

- факт доставки
- первое видимое follow-up действие после доставки
- подтверждение проблемы пользователем, если оно есть

Что важно:

- true read receipts Telegram Bot API не даёт
- значит мы измеряем не "прочитал", а "после доставки было ли видимое действие"

Зачем нужно:

- понимать полезность мониторингов
- видеть шумные сценарии
- улучшать алерты не по ощущениям, а по реакции

## 3. Что такое базовый контроль и базовый мониторинг

Эти термины нужно понимать не как абстракцию, а как продуктовый preset, который можно включить почти без настройки сразу после добавления точки.

### 3.1 Базовый контроль

Базовый контроль — это минимальный готовый контур, который даёт пользователю ощущение, что точка больше не "без присмотра".

Он отвечает на три вопроса:

- есть ли сейчас явная проблема
- что из важного изменилось
- если проблема повторится, придёт ли сигнал автоматически

### 3.2 Базовый мониторинг

Базовый мониторинг — это конкретный набор настроек по умолчанию для одной точки.

Предлагаемый состав:

- `stoplist` monitor
- `blanks` monitor

Предлагаемые дефолтные параметры:

- интервал: `180` минут, если пользователь не указал свой
- рабочее окно: `10:00-22:00` по `Asia/Yekaterinburg`, если пользователь не задал другое
- доставка: в report chat, при необходимости с direct fallback

Контракт по `stoplist`:

- мониторинг должен показывать, что новое появилось, что осталось в стопе и что ушло
- для позиций в стопе должен быть виден возраст позиции
- цветовая логика должна быстро отличать недавно появившиеся позиции от затянувшихся

Контракт по `blanks`:

- если пользователь не указал период для разового запроса, дефолт — последние `3` часа
- жёлтые зоны не считаются красным alert
- alert отправляется только на реальные красные зоны
- длинный запрос должен получать быстрый спокойный ack до завершения

### 3.3 Что пользователь получает в результате

После включения базового контроля пользователь должен без дополнительного обучения понимать:

- какие две главные проблемы система умеет за него отслеживать
- когда именно он получит alert
- куда этот alert придёт
- как это отключить или изменить обычным текстом
