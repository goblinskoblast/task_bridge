# TaskBridge Roadmap, 2026-04-20

## Product Principle

The product now moves from stabilizing core reports to reducing operator effort.
Priority validation stays tied to `telegram_id=137236883` / prod `user.id=16`.

## Stage 4: Free-text Monitor Control

Estimate: 4-6 iterations.

- Show active monitor summary from natural phrases: "что включено", "какие мониторинги", "что присылаешь".
- Disable monitors by point and type without asking for monitor IDs.
- Change monitor window in free text while preserving the old interval unless a new one is stated.
- Parse natural schedule phrases: "каждые 3 часа", "с 10 утра до 22 вечера", "не присылай".
- Keep default monitor window at `10:00-22:00` Asia/Yekaterinburg when the user does not specify a window.

## Stage 5: Scenario Precision

Estimate: 3-5 iterations.

- Align one-off reports and monitor setup for stoplist, blanks, and reviews.
- Make "all saved points" and "one point" behavior consistent across report types.
- Keep blanks default period at the last 3 hours when the user does not specify a period.
- Preserve fast neutral acknowledgements for long-running requests.

## Stage 6: Operator Observability

Estimate: 3-5 iterations.

- Keep strengthening `scripts/inspect_monitor_incident.py` for quick prod analysis.
- Add priority-user attention telemetry: report post -> first visible reaction latency.
- Track monitor delivery chat/message IDs for future correlation.
- Track visible reply/forward metadata in report chats.
- Clearly separate actual Telegram limitations from product bugs.

Important limitation: regular Telegram Bot API does not expose true read receipts for outgoing bot messages. The practical metric is "first action visible to the bot after delivery", not real `seen`.

## Stage 7: Final UX Cleanup

Estimate: 2-3 iterations.

- Remove remaining legacy quick-button hints that no longer match free-text-first UX.
- Keep Telegram screens short, direct, and free of technical details.
- Add golden regression checks around priority user flows.

## Stage 8: Wider Growth Planning

Estimate to plan: 1-2 iterations after stages 4-7.

- Define which new systems and report types matter after Italian Pizza is stable.
- Decide whether TaskBridge becomes mostly an operator copilot, a monitoring console, or a multi-system reporting layer.
- Add metrics for product adoption: active monitor usage, alert usefulness, response speed, and repeated manual questions.

## Current Total Estimate

Stages 4-7: 12-19 iterations.
If we keep tight focus and no major prod incident appears, the aggressive target is about two working days of concentrated iterations.

## Current Status

- Stages 4-7 are now shipped on `main` and backed by priority-user golden regressions.
- The next active planning horizon is `Stage 8: Wider Growth Planning`.
