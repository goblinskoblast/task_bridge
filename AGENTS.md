# TaskBridge Agent Guardrails

## Главный принцип

Работать не реактивно по одному багу за раз, а через продуктовые контракты, регрессии и короткий safety-check соседних сценариев.

## Приоритет продукта

- Главный продуктовый пользователь: `telegram_id=137236883`
- Тестовый пользователь `873075246` используется только для smoke, не как основной ориентир
- Все критичные сценарии и решения сначала проверять относительно приоритетного пользователя

## Нельзя забывать

- Не считать задачу закрытой только потому, что ушёл один симптом
- Каждый найденный продовый баг превращать в регрессионный тест
- После фикса проверять не только исходный кейс, но и соседние сценарии
- Не оставлять пользователю технические причины ошибок, трассировки, mojibake и внутренние детали

## Обязательные продуктовые инварианты

### Telegram UX
- Если запрос может выполняться долго, бот обязан быстро отправить спокойное подтверждение принятия запроса
- Ответы пользователю должны быть простыми, короткими и без внутренней кухни
- Интерфейс должен быть максимально линейным, без лишних кнопок и развилок

### Intent parsing
- Нужно чётко различать:
  - разовый отчёт
  - включение мониторинга
  - включение мониторинга с интервалом
  - отчёты по одной точке и по всем сохранённым точкам
- Формулировки вида `присылай`, `каждые 3 часа`, `каждый день`, `мониторь` должны трактоваться как настройка мониторинга, а не как разовый отчёт

- For `blanks` and `stoplist`, if the user explicitly says `prisylay`, `monitor`, `sledi`, or `uvedomlyay` without an interval, default interval = `180` minutes.
- For `blanks` and `stoplist`, free-text phrases like `не присылай`, `отключи`, `выключи`, and `останови мониторинг` should disable the existing monitor by point without asking for monitor ID.
- If the user changes only the monitor window in free text, keep the existing interval unless a new interval was stated explicitly.

### Blanks
- Для запроса по всем добавленным точкам, если период не указан, дефолт = последние `3 часа`
- Жёлтые / warning / лимитные состояния не должны называться красными зонами
- Красные бланки — это alert-событие
- Логика цветов должна быть устойчивой и не плавать от одной формулировки к другой

### Stoplist
- Free-text запросы по стоп-листу не должны уезжать в browser/general сценарии
- Для приоритетного пользователя сценарий по `Сухой Лог, Белинского 40` считается золотым регрессионным кейсом

### Monitor flow
- Free-text phrases like `покажи мониторинги`, `какие мониторинги`, `активные мониторинги`, `что у меня включено` must show the active monitor summary without requiring `/monitors` or monitor IDs.
- После каждого запуска мониторинга должны корректно обновляться:
  - `last_checked_at`
  - `last_status`
  - `last_result_json`
  - `last_alert_hash`
- Нужно различать:
  - плановый отчёт
  - alert
  - отсутствие события
- Если пользователь включает мониторинг без явного окна, дефолтное окно для продукта: `10:00–22:00` по Екатеринбургу
- Если пользователь задаёт окно в естественной формулировке (`с 10 утра по 22 вечера`), оно должно корректно распознаваться и сохраняться

## Золотые регрессии

Перед тем как считать продовый фикс завершённым, прогнать или проверить максимально близко к продовому поведению:

1. Stoplist по `Сухой Лог, Белинского 40`
2. Blanks по одной точке
3. Blanks по всем сохранённым точкам
4. Включение мониторинга из свободного текста
5. Длинный запрос с быстрым ack
6. Отсутствие user-facing технического мусора

## Definition of Done

Задача считается завершённой только если:

1. Поведение сформулировано как продуктовый контракт
2. На баг добавлен или обновлён тест
3. Проверены соседние риски
4. Если change продовый — проверены логи, monitor state или фактический след в prod
5. Пользовательский текст не ухудшен
6. В отчёте коротко указано:
   - что найдено
   - что исправлено
   - что осталось риском

## Временная зона и контекст

- Пользовательский контекст времени: `Asia/Yekaterinburg`
- Если в инфраструктуре используется другая серверная зона, это нужно явно учитывать и не путать с пользовательским временем

## Стратегия разработки

- Меньше разовых патчей, больше устойчивых контрактов
- Меньше рассуждений ради рассуждений, больше прямой проверки логов, БД и продового следа
- Если проблема повторяема, нужно чинить класс проблемы, а не только текущий симптом

## Текущий архитектурный курс

- Ближайший уровень развития проекта — универсализация TaskBridge, а не только углубление одного `Italian Pizza` контура
- Главный боевой anchor остаётся тот же: `telegram_id=137236883`, его сценарии по-прежнему нельзя ломать
- Следующие целевые системы для каркаса и последующей боевой интеграции:
  - `iiko`
  - `keeper`
- Следующий клиентский канал после Telegram:
  - `MAX`
- После каркаса систем и каналов следующим крупным слоем идут:
  - статистика
  - аналитика по статистике
  - приложение TaskBridge
  - автоматизация аккаунтов клиентов в мессенджерах через агентные контуры

## Правила универсализации

- Новые системы нельзя добавлять как россыпь ad-hoc `if` по строкам и доменам; сначала нужен явный каталог систем, capability contract и слой распознавания
- Перед добавлением нового адаптера нужно зафиксировать:
  - как система определяется
  - какие у неё entry surfaces
  - умеет ли она scan / points discovery / reporting / monitoring
  - какие риски по auth, rate limits и reliability
- Сначала строить каркас `system scan + orientation`, потом подключать конкретный боевой сценарий
- При расширении на новые системы и каналы нельзя ухудшать текущий UX для priority user и текущий `Italian Pizza` поток
- Временные решения по новой системе допустимы только если они не мешают последующей платформенной сборке

## Правила расширения по каналам

- Telegram остаётся основным продовым каналом до появления parity-контуров в `MAX`
- Новые клиентские каналы должны повторять продуктовый контракт текущего бота:
  - быстрый ack на длинный запрос
  - простой free-text-first UX
  - минимум кнопок
  - отсутствие внутренней кухни в ответах
- Автоматизация аккаунтов клиентов в мессенджерах допускается только как позжеидущий слой и только с audit trail, понятными границами действий и возможностью отката

## Режим исполнения плана

- Разработка должна идти по основному плану развития продукта, а не по набору случайных мелких запросов
- Пока собирается тело продукта, не распыляться на некритичные баги и локальную косметику
- Мелкие недочёты фиксить позже отдельной волной чистки, если они не ломают основной продуктовый поток
- Критичные баги и явные риски для пользователя исправлять сразу, даже если они найдены вне текущего шага плана
- В каждый момент держать видимым курс: какой сейчас главный этап, что уже сделано, что будет следующим шагом
- Текущий главный этап: `platform scaffold for systems and channels`
- Текущий ближайший порядок работ:
  - каталог и контракт систем
  - каркас scan/orientation для новых систем
  - подготовка к `iiko` и `keeper`
  - канал `MAX`
  - затем статистика и аналитика

## Коммуникация по проекту

- В каждом статусном ответе пользователю коротко фиксировать:
  - общий статус проекта
  - что сделано прямо сейчас
  - что будет следующим шагом
